# Tauri Mobile Shell — System Design

**Branch**: `feature/tauri-mobile`
**Author / Date**: hackartist · 2026-05-13
**Status**: design

## Summary

Replace the existing Dioxus native Android build (`dx bundle --platform android` + the `mobile` cargo feature in `app/ratel/`) with a Tauri 2.x native shell that hosts the same `dioxus-web` WASM bundle the browser ships. One frontend codebase, two distribution surfaces: web at `ratel.foundation` and an Android APK signed as `co.biyard.ratel`. The shell is a thin sibling crate; shared request/response DTOs live in `app/ratel/` under a type-only Cargo feature so the two binaries import the same Rust types and cannot drift.

## Motivation

- **Code reuse — one bundle, many shells.** The WASM/JS that browsers load becomes the APK's payload byte-for-byte. No parallel mobile UI codebase. No `mobile` feature gates polluting `cfg` blocks throughout the dioxus-web source.
- **Escape hatch from `dx bundle --platform android`.** The dioxus-native-mobile path required a third compile target in `app/ratel/` (`web`, `server`, `mobile`), an NDK toolchain wired into the Makefile, and a `mobile` feature that touched call sites across the codebase. Tauri replaces all of that with a thin separate crate.
- **Native plugin ecosystem.** Tauri 2.x has battle-tested support for camera, biometric, deep links, secure storage, push, and the updater. Each becomes an `#[tauri::command]` exposed to the WASM bundle over a JSON `invoke()` bridge.

## Non-goals (v1)

- iOS support — Android first, iOS deferred to its own follow-up.
- Push notifications, deep links, biometric auth, auto-update, secure storage — each is a future bridge built on the same interop pattern; v1 ships only one demonstrative bridge.
- Hot-updatable web bundle (CDN-served WASM that updates without an app store release) — v1 ships a frozen snapshot per release.
- Playwright on the APK — Playwright does not drive Android WebViews. v1 ships with a manual smoke-test checklist.
- Multi-device fragmentation testing, performance budgets, APK size gates.

## Architecture

### Distribution model

Bundled / offline-first. The APK contains a frozen snapshot of `dx build --web --release` output. WebView loads from `tauri://localhost` (or `http://tauri.localhost` on Android). The only network dependency is `https://ratel.foundation` for server functions.

Hybrid hot-update (CDN-served bundle) was explicitly rejected in favor of App Store compatibility and simpler ops.

`app-shell` (web bundle, wasm32) and `app-shell-tauri` (native shell, aarch64-linux-android) are two distinct binaries. The native shell imports the web crate **as a type-only path dependency** so DTOs are defined exactly once.

### Cargo features

```toml
# app/ratel/Cargo.toml — [features] section after this work
default = []

# Browser build.
web = ["dep:dioxus-web", "dep:reqwest", ...]

# Web build for the tauri android shell — additive on `web`.
# Enables compile-time gating of tauri interop modules.
tauri-web = ["web", "tauri-types"]

# Type-only — DTOs compile, nothing else. Pulled by app/ratel-tauri.
tauri-types = []

# Axum server build (unchanged).
server = [...]

# (Old `mobile` feature deleted.)
```

`web` and `tauri-web` are not mutually exclusive on paper — `tauri-web` extends `web`. Build the browser with `--features web`; build the tauri payload with `--features tauri-web --no-default-features`.

### Interop pattern

Two layers communicate over a JSON bridge at runtime, not via Rust imports:

```
┌────────────────────────────────────────────────────────────────┐
│ WASM bundle (app-shell, wasm32, --features tauri-web)          │
│                                                                │
│  hook: use_external_url(...)                                   │
│    └─ app/ratel/src/tauri/interop/external_url::open(req)      │
│        └─ dx_eval(include_str!("open.js"))                     │
│            └─ runner.send(serde_json::to_value(req))           │
│                └─ runner.recv::<ExternalUrlResponse>()         │
└────────────────────────────────────────────────────────────────┘
                              │ JSON
                              ▼
┌────────────────────────────────────────────────────────────────┐
│ JS driver (open.js, embedded via include_str!)                 │
│  const req = await dioxus.recv();                              │
│  const res = await window.__TAURI__.invoke(                    │
│      "open_external_url", { req });                            │
│  dioxus.send(res);                                             │
└────────────────────────────────────────────────────────────────┘
                              │ Tauri IPC
                              ▼
┌────────────────────────────────────────────────────────────────┐
│ Tauri shell (app-shell-tauri, aarch64-linux-android)           │
│                                                                │
│  #[tauri::command]                                             │
│  fn open_external_url(req: ExternalUrlRequest)                 │
│      -> Result<ExternalUrlResponse, ExternalUrlError>          │
│      use app_shell::tauri::types::external_url::*;             │
└────────────────────────────────────────────────────────────────┘
```

Both ends import the same `ExternalUrlRequest` / `ExternalUrlResponse` types from `app/ratel/src/tauri/types/external_url.rs`. Any schema change forces both sides to recompile.

This is the existing `dioxus::document::eval` channel pattern documented in `conventions/dioxus-app.md` and used today by `features/auth/interop/wallet_connect.rs`. The only new thing on the JS side is `window.__TAURI__.invoke(...)` enabled by `withGlobalTauri: true` in `tauri.conf.json`.

### Build pipeline

| Target | Command |
|---|---|
| Dev (browser) | `cd app/ratel && dx serve --features web --port 8000` |
| Dev (tauri android) | `cd app/ratel && dx serve --features tauri-web --no-default-features --port 8000`<br>`cd app/ratel-tauri && cargo tauri android dev` |
| Release (browser) | `cd app/ratel && dx build --web --release --features web` |
| Release (tauri APK) | `cd app/ratel-tauri && make build-release` |

The release-APK target script:

1. `cd ../ratel && MOBILE_API_URL=https://ratel.foundation DYNAMO_TABLE_PREFIX=ratel-prod dx build --web --release --features tauri-web --no-default-features`
2. Find the produced `index.html` (dx output dir has shifted across versions — locate dynamically rather than hardcode the path) and rsync the tree into `app/ratel-tauri/dist/`.
3. `cargo tauri android build --target aarch64-linux-android` reads `tauri.conf.json`'s `frontendDist: "../dist"` and produces `src-tauri/gen/android/app/build/outputs/apk/release/app-release.apk`.

Dev iteration: only `#[tauri::command]` or `tauri::Builder` changes trigger a Tauri rebuild. Rust UI edits hot-reload through `dx serve` directly to the WebView via `devUrl`.

### Backend connectivity (cross-origin work)

The WebView origin is `http://tauri.localhost` (Android). The backend is `https://ratel.foundation`. Three fixes:

1. **API base URL baked at build time.** The repo already has `MOBILE_API_URL` in `app/ratel/Makefile` from the dioxus-native-android era. Reuse it. Audit `app/ratel/src/` so every relative `/api/...` call honors the env var when set. The `tauri-web` build always sets it.

2. **CORS + session cookies cross-origin.**
   - Server: `tower-http` CORS layer in `app-shell` allows origin `http://tauri.localhost` with `allow_credentials: true`. Session cookies set with `SameSite=None; Secure; HttpOnly; Path=/`.
   - Client: every WASM `fetch`/`reqwest` from a `tauri-web` build uses `credentials: 'include'`. Single helper; route everything through it.
   - WebView: Android System WebView blocks third-party cookies by default. The tauri shell's `lib.rs` calls `setAcceptThirdPartyCookies(webView, true)` once at init.

3. **No runtime `is_tauri()` detection.** The compile-time `tauri-web` feature replaces it. Browser builds cannot link the tauri code path; tauri builds cannot link the browser fallback.

## v1 verification gate

1. **Browser web build still works** — existing Playwright suite green against `dx serve --features web`. Load-bearing check; everything else is additive.
2. **Tauri-web build compiles** — CI adds `cargo check --features tauri-web --no-default-features` and `dx check --web --features tauri-web --no-default-features` to the dx-check job.
3. **Tauri shell builds APK locally** — `cd app/ratel-tauri && make build-release` produces a debug-keystore-signed APK. Steps documented in `app/ratel-tauri/README.md`.
4. **APK installs and cold-starts on Android emulator** — `make install-debug` to a running Pixel 6 API 34 AVD; WebView loads, dioxus-web hydrates, home screen renders.
5. **Auth round-trip works inside the WebView** — log in with bypass code `000000`, session cookie survives app foreground/background, authenticated routes load. Validates the cross-origin cookie config.
6. **One real native bridge proves the interop layer** — `open_external_url(url: String)` via `tauri-plugin-opener`. DTO in `app/ratel/src/tauri/types/external_url.rs`, command in `app/ratel-tauri/src-tauri/src/commands/external_url.rs`, hook bridge in `app/ratel/src/tauri/interop/external_url/`, one call site in dioxus-web. Verifies the full WASM → eval → invoke → Rust → native → back loop. Every future bridge is the same shape.

Manual smoke-test checklist (lives in `app/ratel-tauri/README.md`):

- [ ] Cold start under 3s on Pixel 6 AVD
- [ ] WASM bundle loads, hydration completes (no errors in WebView devtools console)
- [ ] Sign-in flow completes, session cookie persists across app kill/relaunch
- [ ] Server functions return data on any list view
- [ ] The v1 demonstrative `invoke()` call executes end-to-end
- [ ] Back button closes app cleanly (no JNI crash)
- [ ] No third-party-cookie warnings in `adb logcat`

## Open risks

1. **Tauri WebView cookie storage durability across app cold starts** — third-party cookies in Android System WebView are technically supported but have a history of inconsistency. If session cookies are lost across cold starts, we fall back to a bearer-token migration (login returns a JWT, stored in tauri's keyring, attached as `Authorization` header). This is the contingency, not the v1 plan.
2. **WebView CSP and Tauri's IPC channel coexistence** — the CSP in `tauri.conf.json` must allow `tauri:` schemes and `https://ratel.foundation` while still being restrictive enough for review. Will require iteration against logcat.
3. **dx build output path drift between dioxus versions** — handle via `find` in the build script rather than a hardcoded path.
4. **`withGlobalTauri: true` exposes `window.__TAURI__` to all loaded code.** Acceptable because the bundle is shipped by us, but worth noting if we ever load third-party iframes.
